# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

**Brianna** — Manager dashboard for a trucking company to oversee team spending, approve purchase requests, monitor budget, and generate reports. Built for Brim Financial x MPC Hacks 2026. See `docs/PRD.md` for full requirements.

Two roles switched via a UI toggle (no auth): **Employee** (submit requests) and **Manager** (full dashboard). Single-manager scope — no multi-department view.

## Commands

```bash
# Backend (Express + TypeScript, runs on http://localhost:3001)
cd backend
npm install
cp .env.example .env   # add ANTHROPIC_API_KEY
npm run dev            # ts-node with nodemon watch (src/**/*.ts)
npm run build          # compile TypeScript → dist/
npm start              # run compiled dist/index.js

# Frontend (React + Vite, runs on http://localhost:5173)
cd frontend
npm install
npm run dev
npm run build
npm run lint

# Data
python data/process_data.py   # regenerate transactions.xlsx from raw source
```

Node >=22 is required — the backend uses `node:sqlite` (built-in, experimental).

## Architecture

Monorepo with two packages:

```
backend/     — Node.js + Express + TypeScript API server
frontend/    — React + Vite + TypeScript UI
data/        — transactions.xlsx dataset + process_data.py transform script
docs/        — PRD.md (requirements), issues_backlog.md (22 issues)
```

### Data layer

`data/transactions.xlsx` (4,235 rows, 16 columns) is loaded at backend startup into a **SQLite in-memory database** (`node:sqlite` built-in, `new DatabaseSync(':memory:')`). No persistent database — data reloads from Excel on each server restart. All analytical queries are SQL generated by Claude and executed against this in-memory DB.

**Transactions table columns** (snake_case; Excel header → DB column mapping in `db/index.ts`):
`transaction_code`, `transaction_description`, `transaction_category`, `posting_date`, `transaction_date`, `merchant_name`, `amount`, `debit_or_credit`, `merchant_category_code`, `merchant_city`, `merchant_country`, `merchant_postal_code`, `merchant_state`, `conversion_rate`, `employee_name`, `category_label`

Excel headers differ from DB names for most columns (e.g. `Transaction Amount` → `amount`, `Merchant Info DBA Name` → `merchant_name`, `Posting date of transaction` → `posting_date`). Dates are stored as ISO strings (YYYY-MM-DD), converted from Excel serial numbers. `debit_or_credit` is lowercased on import — always filter with `'debit'` not `'Debit'`.

**Requests table** (session-only, created at startup):
`id` (uuid), `employee_name`, `item_description`, `amount`, `category`, `reason`, `status` (pending/approved/denied), `created_at`

Key data facts:
- `employee_name` — 8 rotating fake names: Jordan, Maya, Tyler, Priya, Marcus, Sofia, Ethan, Leila
- `category_label` — human-readable MCC label. Key mappings: Fuel (5541/5542), Meals (5812), Vehicle Maintenance (7538), Government Permits (9399), Parts & Supplies (5046), Tolls & Permits (4784)
- Total budget constant: **$50,000** — defined in `backend/src/lib/config.ts` as `TOTAL_BUDGET`
- Only `debit` transactions count toward spend totals; filter `WHERE debit_or_credit = 'debit'`

### AI layer

All Claude calls are server-side only — never from the frontend. Use `claude-sonnet-4-6` as the default model. Each feature uses a multi-step reasoning chain:

- **F0 (Budget Tracker):** SQL aggregation → total spend vs budget gauge + category pie chart
- **F1 (Chat):** intent extraction → Claude generates SQL → SQLite query → contextual analysis → visualization selection
- **F2 (Compliance):** context gathering → policy comparison → severity scoring (Critical/High/Medium/Low)
- **F3 (Request Flow):** employee submits form → Claude generates approve/deny/escalate recommendation with reasoning → manager one-click decision → budget updates in real time
- **F4 (Reports):** Period Report (weekly/monthly exec memo) or Employee Report (spend profile vs team average)

Every Claude response returning structured data is validated with **Zod**. Retry once on parse failure; return a structured error on second failure.

Reusable wrapper: `backend/src/lib/claude.ts` (implemented):
- `askClaude<T>(prompt, schema, options?)` — `options: { model?, maxTokens?, systemSuffix? }`. Defaults: `claude-sonnet-4-6`, 4096 tokens. `systemSuffix` appends to the base "respond with valid JSON" system prompt.
- `ClaudeError = { error: string; raw: string }` — thrown on second parse failure. Route handlers should catch this type and return an HTTP error.

### Frontend tech

- **Tailwind v4**: no `tailwind.config.js`. Plugin loaded via `@tailwindcss/vite` in `vite.config.ts`. Only entry needed in `src/index.css`: `@import "tailwindcss"`.
- **ESLint**: type-aware (`tseslint.configs.recommendedTypeChecked`). `parserOptions.project: true` in `eslint.config.js` — running `npm run lint` requires a valid `tsconfig.app.json`.
- **Vite proxy**: all `/api/*` requests forwarded to `http://localhost:3001` — no CORS headers needed on the backend in dev.

### Security

`ANTHROPIC_API_KEY` lives only in `backend/.env`. Never pass it to or call the Anthropic API from the frontend.

Health check: `GET http://localhost:3001/api/health` → `{ status: "ok" }`

Debug endpoint (created with data layer): `GET /api/debug/transactions?limit=5`

## Key Decisions

- **`node:sqlite` (built-in) over `better-sqlite3`** — avoids native compilation. Import: `import { DatabaseSync } from 'node:sqlite'`. Synchronous API — no async needed. Still experimental in Node v24 but stable enough for this project.
- **No persistent database** — data reloads from `transactions.xlsx` on each server start; `requests` table is session-only
- **Role toggle, not auth** — `useState` in React switches between Employee and Manager views; no sessions or passwords
- **Claude over Gemini** — better structured JSON output and tool use reliability
- **Split-charge detection in SQL** — same employee, amounts within 10% of each other, same merchant, within 48h; runs independently of Claude
- **Claude-generated SQL is read-only** — never allow INSERT/UPDATE/DELETE from Claude-generated SQL in the chat chain
- **Vite proxy** — `frontend/vite.config.ts` proxies `/api` → `http://localhost:3001` to avoid CORS in dev
